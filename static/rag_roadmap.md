# RAG 项目企业级扩展方案

> 目标：从能跑通的 RAG Demo，演进为企业级可用的智能文档问答系统。
> 每个模块标注优先级：🔴 高优先级 🟡 中优先级 🟢 低优先级

---

## 一、文档处理优化

### 🔴 1. 文档处理异步化（Celery）

**现状问题**
文档上传后，解析 + 切片 + Embedding 是同步阻塞的，大文件可能耗时几十秒，接口直接超时。

**企业级做法**
上传接口只负责接收文件、创建记录，处理任务丢给 Celery 异步执行。

```
上传接口 → 创建记录（status=pending）→ 发送 Celery 任务 → 立即返回
Celery Worker → 处理文档 → 更新状态为 done/failed
前端轮询或 SSE 推送状态变更
```

**面试价值**：异步任务设计、任务状态管理、重试机制。

---

### 🔴 2. 文档解析增强

**现状问题**
- PDF 解析只提取文本，表格、图片内容丢失
- 不支持 Markdown、HTML、Excel 等格式

**企业级做法**
- 接入 `unstructured` 库，支持更多格式和结构化解析
- 支持 Markdown、HTML、Excel、CSV
- 表格内容转为文本描述后一并存入向量库

**面试价值**：多格式文档处理、非结构化数据处理能力。

---

### 🟡 3. 切片策略优化

**现状问题**
所有文档用统一的 chunk_size=500，没有针对不同文档类型优化。

**企业级做法**
- 代码文档：按函数/类边界切片
- Markdown：按标题层级切片
- 普通文档：当前策略 + 语义切片（按句子边界）
- 支持在上传时指定切片策略

**面试价值**：切片策略选型、不同场景的 trade-off。

---

## 二、检索质量优化

### 🔴 4. 混合检索（Hybrid Search）

**现状问题**
纯向量检索对精确关键词（人名、型号、专有名词）效果差。比如用户问"XR-500 的参数是多少"，向量检索可能找不准。

**企业级做法**
向量检索 + BM25 关键词检索，结果用 RRF（倒数排名融合）算法合并：

```
用户问题
  ├── 向量检索 → Top-K 结果
  └── BM25 检索 → Top-K 结果
        ↓
      RRF 融合排序
        ↓
      最终 Top-K
```

**面试价值**：混合检索原理、召回率 vs 精确率的平衡。

---

### 🔴 5. Rerank 重排序

**现状问题**
检索返回 k=4 个结果，但这 4 个不一定都和问题高度相关，有噪音。

**企业级做法**
先用向量检索召回更多候选（k=20），再用 Rerank 模型精排，取 Top-4：

```
向量检索 Top-20
  → Rerank 模型（bge-reranker-base）重新打分排序
  → 取 Top-4 注入 Prompt
```

**面试价值**：两阶段检索、召回与精排的分工。

---

### 🟡 6. 检索结果过滤与去重

**企业级做法**
- 相似度分数过滤：低于阈值的切片直接丢弃，避免噪音
- 语义去重：高度相似的切片只保留一个
- metadata 过滤：支持按文档类型、时间范围过滤检索范围

## 三、对话与记忆优化

### 🔴 7. Token 数限制的上下文管理

**现状问题**
历史消息按条数截断（最近 10 条），不够精确，可能截掉重要上下文。

**企业级做法**
按 Token 数动态管理历史：

```python
def trim_history_by_tokens(history, max_tokens=3000):
    result = []
    total = 0
    for msg in reversed(history):
        tokens = estimate_tokens(msg.content)
        if total + tokens > max_tokens:
            break
        result.insert(0, msg)
        total += tokens
    return result
```

---

### 🟡 8. 对话摘要记忆（Summary Memory）

**现状问题**
历史消息超过限制直接丢弃，早期对话内容丢失。

**企业级做法**
超过 token 限制时，把早期对话压缩成摘要，用摘要替代原始消息：

```
[摘要：用户询问了文档第一章的主要内容，助手回答了...] 
+ 最近 N 条完整消息
```

**面试价值**：LangChain Memory 机制、长对话的上下文压缩策略。

---

### 🟡 9. 多文档联合检索

**现状问题**
每次只能针对单个文档提问。

**企业级做法**
支持选择多个文档，跨文档检索，在多个 collection 里同时搜索后合并结果。

## 四、Agent 化改造

### 🔴 10. 基础 Agent 改造（Tool Use）

**现状问题**
现在是固定的 RAG 流程：每次提问都强制检索文档，即使问题和文档无关（比如"你好"）也会检索，浪费 token。

**企业级做法**
用 LangChain Agent 替换固定流程，让 LLM 自主决策：

```python
tools = [
    search_document_tool,   # 检索文档
    get_current_time_tool,  # 获取当前时间
    calculator_tool,        # 计算器
]
# LLM 自己决定：这个问题需要检索吗？需要调用哪个工具？
```

**面试价值**：Agent vs RAG 的区别、Tool Use 设计、ReAct 模式。

---

### 🟡 11. LangGraph 多步 Agent

**现状问题**
基础 Agent 是单轮工具调用，复杂问题（需要多步推理）处理能力不足。

**企业级做法**
用 LangGraph 构建有状态的 Agent 工作流：

```
用户问题
  → 路由节点（判断问题类型）
  → 检索节点（如需要）
  → 推理节点
  → 验证节点（答案是否基于文档）
  → 输出节点
```

**面试价值**：LangGraph 工作流编排、复杂 Agent 设计。

## 五、工程质量优化

### 🔴 12. 可观测性（LLM Tracing）

**现状问题**
LLM 调用是黑盒，出问题不知道发的是什么 Prompt，检索了哪些内容，耗时多少，花了多少 token。

**企业级做法**
每次 LLM 调用记录完整链路日志：

```
请求 ID、session_id、document_id
检索到的切片内容
完整 Prompt
LLM 回答
耗时、token 消耗、模型名称
```

存入 PostgreSQL，提供查询接口，方便排查问题。

**面试价值**：LLM 应用的可观测性设计，这是企业级必备。

---

### 🔴 13. 接口限流与成本控制

**现状问题**
无任何限流，用户可以无限调用，LLM API 费用不可控。

**企业级做法**
- 基于 Redis 的滑动窗口限流：每个用户每分钟最多 10 次问答
- Token 预算控制：单次请求 token 上限
- 每日用量统计与告警

**面试价值**：限流算法、成本控制，这是企业项目必须考虑的。

---

### 🟡 14. 文档处理队列优先级

**企业级做法**
Celery 支持任务优先级，VIP 用户的文档处理优先于普通用户。

---

### 🟡 15. 健康检查与监控

**企业级做法**
- `/health` 接口：检查 PostgreSQL、Redis、ChromaDB、LLM API 是否正常
- 关键指标监控：文档处理成功率、平均检索耗时、LLM 平均响应时间
- 异常告警

---

## 六、用户体验优化

### 🟡 16. 流式状态推送

**现状问题**
文档处理是异步的，用户上传后不知道处理进度。

**企业级做法**
用 SSE 或 WebSocket 实时推送文档处理状态：
```
上传完成 → 解析中（20%）→ 切片中（50%）→ 向量化中（80%）→ 完成
```

**面试价值**：你已有 SSE 实战经验，直接可以用上。

---

### 🟡 17. 答案来源引用

**现状问题**
LLM 回答了一段话，用户不知道这个答案来自文档的哪个部分。

**企业级做法**
在回答后附上来源切片：
```
回答：...

参考来源：
[1] 第3页：...原文片段...
[2] 第7页：...原文片段...
```
